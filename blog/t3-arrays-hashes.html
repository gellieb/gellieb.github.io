What's the difference between arrays and hashes?

January 8, 2015

Hello!


Today's blog is comparing and contrasting Arrays and Hashes. They are similar in many ways, but have structural and functional differences to distinguish the two. Let's start of with arrays.



Ruby-Doc describes arrays as "ordered, integer-indexed collections of any object. Array indexing starts at 0..." Okay, so an Array is basically a collection of strings, integers, symbols, etc, or a combination of any of these. Each item, aka element, in the array has an index which dictates its position in the array and also starts from 0. The great thing about arrays is that I can call any element in the array if I know it's position. To do that, we'll do: 

puts array[#]

The # is the index of the element. The index increases sequentially, so from 0, 1, 2, 3....the last element's position. So, array's are pretty straight-forward.



Hashes, on the other hand, in Ruby-Doc, is described as "a *dictionary-like collection* of unique keys and their values... they are similar to Arrays, but where an Array uses integers as its index, a Hash allows you to use any object type."

This is a pretty apt definition of what a Hash is. Similar to an Array's indexing, which by the way is only numerical (0-#), Hashes have keys. Keys can be anything-- a string, a number, a symbol, and even an array-- and they must always have a value (hence, key-value pair)! The benefit of Hash's keys is that they don't have to be numerical and can, therefore, have a name that reflects that its values are.

hash = { :first_name => "Angel", :last_name => "Baek"}


Similar to an Array, to call the value "Angel", do: 

puts hash[:first_name]


You can see that an Array's index is equivalent to a Hash's key. Just to reiterate, a key must always have a value! You can set to nil or another specified value but just remember a key needs its pair.



To ponder the uses, I'll compare Arrays and Hashes to the shelves of a refrigerator. Remember how an Array is just a collection of stuff? It can even contain arrays as its elements. So an array is like the fruit basket container at the base of the fridge-- you can have different, but ordered, compartments in there of fruits and veggies; however, if you start to include containers containing other containers (nested arrays) inside that fruit basket compartment, things might get hairy and difficult to sort.



A Hash's keys are like the different shelves/levels of the refrigerator:


1. Each shelf is unique == keys can have any object as their names
2. There can be one item or many different items per shelf == keys can store a value that is a single object 

{:first_name => "Angel"}

3. or be an array containing multiple elements 

{:girl_names=> ["Jessica", "Kimberly", "Eloise"]}

4. ...that can include sub-shelves/containers 

{:names=>
        [{:girl_names=> ["Jessica", "Kimberly", "Eloise"]},
        {:boy_names =>["Jacob", "Timothy", "Brandon"]}]}



Unlike the fruit basket analogy, containers separating different foods on the same level in fridge isn't very complicated or difficult to manage. Ultimately, it comes down to data storage and preference and legibility. Hashes definitely trumps Arrays in how much they can store.

I hope that cleared some ambiguity between the two!
